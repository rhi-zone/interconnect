# Agent Orchestration

AI coding assistants — Claude Code, Gemini CLI, and similar tools — are short-lived processes. They start, answer a prompt, and exit. Rooms, by contrast, are persistent: messages arrive at any time, connections stay open, history accumulates.

The daemon bridges these two models. It holds room connections across assistant invocations. Hooks in the assistant's configuration inject incoming messages into context at natural pause points and route outgoing messages to rooms when the assistant finishes.

The result: the assistant participates in rooms without knowing anything about routing. Messages come in; responses go out. The plumbing is outside the assistant's concern.

## The Setup

Three components:

1. **`interconnect-daemon`** — runs in the background, holds connections to all configured rooms.
2. **`.claude/settings.json`** (or equivalent) — hooks generated by `interconnect init --preset claude` that wire assistant lifecycle events to `interconnect` CLI calls.
3. **`INTERCONNECT_REPLY_TO`** — environment variable naming the room that should receive the assistant's output.

## The Full Loop

### 1. Start the daemon

```sh
interconnect-daemon &
```

The daemon reads `interconnect.toml`, connects to all configured rooms, and listens on `~/.interconnect/daemon.sock`.

### 2. Generate hooks

```sh
interconnect init --preset claude --output .claude/settings.json
```

This reads `interconnect.toml` and writes a `.claude/settings.json` with hooks attached to Claude Code's lifecycle events. Run this once per project after changing your room configuration.

### 3. Invoke the assistant with a reply target

```sh
INTERCONNECT_REPLY_TO=work-chat claude -p "summarize the last hour of discussion"
```

The hooks fire automatically during the session:

- Before each tool call and before each prompt is processed: pending messages from `work-chat` are injected into context.
- After each tool call: the tool event JSON is sent to `work-chat`.
- When Claude stops: the final response is sent to `work-chat`.

### 4. React to incoming messages

```sh
interconnect watch work-chat --exec 'INTERCONNECT_REPLY_TO=work-chat claude -p "$(cat)"'
```

`watch` blocks on `work-chat`. When a message arrives, it pipes the message JSON to the shell command with `INTERCONNECT_REPLY_TO` already set. Claude wakes, reads the message as its prompt, and the hooks route its response back to the room.

This is the reactive pattern: the room wakes the assistant; the assistant replies to the room; the loop continues.

## What the Hooks Do

The Claude preset generates four hook entries:

### `PreToolUse` — inject pending messages

Before Claude executes a tool, the hook calls:

```sh
interconnect recv --nowait "$INTERCONNECT_REPLY_TO" 2>/dev/null || true
```

If there are unread messages waiting in the room, they are printed to stdout — Claude Code shows hook stdout to the assistant before proceeding. If there are no pending messages, the command prints nothing and Claude continues normally.

This is ambient injection: the assistant does not poll or wait. Messages arrive in context when the assistant is already paused at a natural boundary.

### `UserPromptSubmit` — inject pending messages

Same command as `PreToolUse`, run when a user prompt is submitted. Ensures that messages that arrived since the last tool call are visible before Claude processes the new prompt.

### `PostToolUse` — log tool events

After each tool call, the hook forwards the event JSON (piped from stdin by Claude Code) to the reply-to room:

```sh
interconnect send "$INTERCONNECT_REPLY_TO" "$(cat)" 2>/dev/null || true
```

This produces a running log in the room: every file read, every bash command, every search — not just the final summary. Useful for audit trails, debugging, and observer processes that want to follow the assistant's work in real time.

### `Stop` — send final response

When Claude finishes its turn, the hook sends a stop notification:

```sh
interconnect send "$INTERCONNECT_REPLY_TO" '{"type":"stop"}' 2>/dev/null || true
```

This signals that the assistant has completed. A `watch` loop or another process listening on the room knows the turn is done.

All hooks suppress errors (`2>/dev/null || true`). A missing daemon, an unset `INTERCONNECT_REPLY_TO`, or a network hiccup does not interrupt the assistant's normal operation.

## Example `interconnect.toml` for an Agent Setup

```toml
# The room the agent reads from and writes to
[[room]]
name       = "work-chat"
connector  = "slack"
bot_token  = "xoxb-..."
app_token  = "xapp-..."
channel_id = "C1234567890"

# A local session log — captures everything the agent does
[[room]]
name = "session"
connector = "sqlite"
path = ".interconnect/session.db"

[room.chat_log]

[room.chat_log.columns]
timestamp = { path = "timestamp",  type = "integer" }
event     = { path = "type",       type = "text" }
raw       = { path = "*",          type = "json" }
```

After writing this file, regenerate the hooks:

```sh
interconnect init --preset claude --output .claude/settings.json
```

Then start a watch loop that routes Slack messages to the agent and the agent's replies back to Slack:

```sh
interconnect watch work-chat --exec 'INTERCONNECT_REPLY_TO=work-chat claude -p "$(cat)"'
```

## Sending Messages Manually

The `watch` loop handles the reactive case. For one-off invocations, set `INTERCONNECT_REPLY_TO` directly:

```sh
INTERCONNECT_REPLY_TO=work-chat claude -p "what changed in the last commit?"
```

Or read pending messages explicitly before starting:

```sh
interconnect recv work-chat   # block until a message arrives
INTERCONNECT_REPLY_TO=work-chat claude -p "$(interconnect recv --nowait work-chat)"
```

## Other Tools

The preset system is tool-agnostic. The daemon knows nothing about Claude, Gemini, Cursor, or any other assistant. It speaks `interconnect recv`, `interconnect send`, and `interconnect watch`.

Other tools follow the same pattern:
- **Gemini CLI**: equivalent hook configuration in its settings file, same CLI commands.
- **Custom scripts**: call `interconnect recv --nowait` to pull pending messages, `interconnect send` to post results.

`interconnect init --preset <name>` generates the tool-specific hook file. The underlying commands are identical across all presets.

## Design Notes

The assistant is not an authority in the Interconnect sense — it does not own a room. It is a client that reads from and writes to rooms owned by other authorities (Slack, Discord, a SQLite file). The daemon manages those connections; the assistant just participates.

The hook interface (`PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Stop`) is converging across major AI coding tools. Presets are templates over this shared interface — one per tool, community-extensible, no daemon changes required.
